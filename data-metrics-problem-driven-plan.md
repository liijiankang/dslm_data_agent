# Data Metrics 问题驱动指标建设方案

版本：v5

## 1. 本次改造重点

这版方案在 v3 的前台体验基础上，重点升级后台资产关系：前台继续保持轻量确认，后台从刚性一对一升级为“主资产默认、扩展/共享建议”，以兼顾治理清晰度、分析深度和计算效率。

| 改造点 | 方案落位 |
| --- | --- |
| 弱化指标集创建 | 用户确认分析方向即可，系统后台自动生成对应指标集；前台不再把“分析方向”和“指标集”拆成两个确认动作 |
| 弱化指标体系层级 | 指标体系降级为可选的场景标签或资产分组，不再要求用户在生成流程中强制选择或创建 |
| 拆解框架改为解释层 | 系统仍生成拆解框架，但作为“推荐依据”折叠展示在分析方向页面，不再作为独立强制步骤 |
| 支持延迟澄清 | 意图草案不要求一次性补齐全部口径，允许在分析方向和指标确认阶段继续补充澄清 |
| 区分核心和扩展方向 | 分析方向分为核心方向、扩展方向和暂缓方向，避免一次性平铺过多推荐 |
| 主资产默认 | 每个分析方向默认有唯一主指标集、主逻辑输出资产和主计算入口，保证治理锚点清晰 |
| 扩展指标集 | 当指标分层明显或消费场景不同，系统建议拆出扩展指标集，不要求用户主动配置 |
| 共享物理资产 | 多个分析方向源表、粒度、调度和权限高度一致时，系统建议共享物理输出资产，减少重复建设 |
| 共享或合并计算任务 | 多个计算任务重叠度高时，系统建议合并计算链路，同时保留方向级监控和告警视角 |
| 存量资产冲突和去重 | 在意图确认后先做资产扫描，并在分析方向、指标、维度、指标集和输出资产确认时持续提示复用、合并、新建版本或冲突审核 |
| 方案草稿、协同和版本 | 将“指标建设方案”定义为可持久化对象，支持草稿恢复、协同编辑、审核意见、版本对比和回滚 |
| 下游消费反向约束 | 增加口径变更、影响分析、历史数据回刷和下游通知机制 |
| 指标语义层 / Metric DSL | 增加机器可执行、可版本化、可测试、可编译 SQL 的指标定义协议 |
| 指标类型体系 | 区分原子、派生、复合、比较、诊断、质量和运营阈值指标，避免不同计算逻辑混用 |
| 可加性和聚合治理 | 增加完全可加、半可加、不可加，以及默认聚合、时间聚合和粒度约束 |
| 冲突判断规则 | 用名称、业务定义、计算表达式、来源字段和过滤条件做相似度评分，驱动复用、合并、版本和人工确认 |
| 数据模型推荐可信度 | 数据模型不只展示名称，还展示覆盖度、可信度、缺失字段、数据新鲜度和质量风险 |
| 维度分层治理 | 区分公共维度和方向特有维度，公共维度在同一业务主题或场景标签内保持统一口径 |
| 输出资产扩展 | 当前默认 ADS 结果表，未来预留查询视图、API 服务、BI 虚拟数据集和预聚合缓存 |
| 业务主题跨域 | 一个方案有主业务主题，也可以关联其他业务主题，实现跨主题资产复用 |
| AI 原生交互 | 初步澄清后生成“意图结构化草案”和“口径定义草案”，并在方向确认、指标确认阶段持续完善 |

改造后的核心变化是：前台让用户围绕“业务问题、业务主题、分析方向、维度指标、语义定义、逻辑输出资产”做确认；后台通过 Metric DSL 自动维护主资产、扩展资产、共享物理资产、版本、血缘、测试和 SQL 编译，避免把治理概念全部推给用户。

## 2. 产品定位

Data Metrics 是 AI 原生指标平台，核心目标不是临时回答业务问题，而是把业务问题转化为可治理、可复用、可发布的指标资产。

用户在 Data Metrics 中提出问题，本质上不是发起一次分析问答，而是用自然语言表达一次指标建设需求。临时问答、归因解释、洞察总结更适合放在 Data Analytics 中完成；Data Metrics 负责沉淀稳定的指标资产，供 Data Analytics、BI、API、问数服务和其他下游场景消费。

一句话定位：

> Data Metrics 不直接回答问题，而是把问题沉淀为业务主题、分析方向、指标、维度、Metric DSL、逻辑输出资产和计算配置，并在后台形成主指标集、扩展指标集、共享物理资产、标签、版本、血缘、测试和共享计算链路等治理资产。

## 3. 核心设计原则

### 3.1 先理解业务意图，再反推数据模型

问题驱动流程应该从业务视角出发，而不是从数据模型视角出发。

推荐顺序是：

```text
业务问题
  -> 初步澄清用户意图
  -> 生成初版意图结构化草案
  -> 推荐主业务主题和关联业务主题
  -> 推荐分析方向
     -> 折叠展示推荐依据
     -> 支持方向级补充澄清
  -> 按分析方向推荐数据模型
  -> 按分析方向确认维度和指标
     -> 支持指标级补充澄清
  -> 按分析方向生成逻辑输出资产和计算配置
  -> 发布并进入治理
```

这样更符合 AI 原生指标平台的定位：

```text
业务意图 -> 业务主题 -> 分析方向 -> 数据模型 -> 指标/维度 -> 逻辑输出资产 -> 计算配置
```

### 3.2 分析方向是建设和计算落地的边界

一个业务问题通常是复合问题，会被拆成多个分析方向。

当前版本建议把分析方向作为用户侧的核心建设边界：

```text
一个分析方向 -> 一组指标和维度
一个分析方向 -> 一组数据模型
一个分析方向 -> 一个逻辑输出资产
一个逻辑输出资产 -> 一个物理输出资产，或共享物理输出资产
一个物理输出资产 -> 一个计算任务，或共享/合并计算任务
```

指标集和物理资产共享关系仍然存在，但作为后台自动维护的治理关系，不需要用户在生成流程中单独创建和配置。

### 3.3 资产感知优先于资产新建

Data Metrics 不是简单生成新资产，而是应当具备资产感知能力。

在推荐业务主题、分析方向、指标、维度、指标集和输出资产时，系统都应先检查存量资产，判断是否已有可复用资产、相似资产、冲突口径或历史版本。

系统推荐行为不应只有“新建”，还应包括：

- 复用已有
- 合并补充
- 基于已有新建版本
- 标记冲突待审核
- 放弃推荐

### 3.4 用户信任来自可解释的 AI 推荐

AI 不应只给出结论，还应解释为什么这么拆、为什么推荐这些数据模型、为什么推荐这些指标和维度。

尤其是“分析方向推荐”不能成为黑盒。系统应在分析方向列表上方提供可折叠的“推荐依据”，例如：

> 针对“成交金额下降”，系统采用“成交金额 = 订单量 × 客单价 × 支付转化率”的拆解框架，并补充退款、渠道、商品结构等外部扰动因素，因此推荐订单趋势、支付转化、渠道贡献、退款影响和商品结构分析方向。

这层解释默认折叠，不作为独立强制确认步骤。专家用户可以直接确认方向，业务用户可以展开查看为什么这样推荐。

### 3.5 主资产默认、扩展/共享建议

当前版本不再采用刚性一对一关系，而是采用“主资产默认、扩展/共享建议”的灵活约束：

```text
一个业务问题 -> 一个指标建设方案
一个指标建设方案 -> 一个主业务主题
一个指标建设方案 -> 多个可选场景标签
一个指标建设方案 -> 多个分析方向
一个分析方向 -> 一个主指标集（后台自动生成，唯一）
一个分析方向 -> 零到多个扩展指标集（系统建议，用户确认）
一个分析方向 -> 一个逻辑输出资产（默认，唯一）
多个逻辑输出资产 -> 可共享一个物理输出资产（系统建议，用户确认）
一个物理输出资产 -> 一个主计算任务（默认，唯一）
多个主计算任务 -> 可共享或合并计算链路（系统建议，用户确认）
```

设计要点：

- 主资产确保每个分析方向有唯一治理锚点和血缘入口。
- 扩展/共享由系统自动检测并建议，不增加用户主动配置负担。
- 扩展指标集必须复用主指标集的业务口径，只在指标分层、消费场景或技术实现上有差异。
- 共享物理输出资产时，各分析方向仍保留独立逻辑输出资产、血缘链路、权限说明和消费说明。
- 共享或合并计算任务时，各分析方向仍保留独立监控、告警和影响分析视角。

这些复杂性默认不暴露给用户，只在系统检测到明显收益时以“接受建议 / 保持独立”的二选一方式出现。

## 4. 核心资产模型

整体关系建议定义为：

```text
业务问题
  -> 指标建设方案
      -> 主业务主题
      -> 关联业务主题
      -> 场景标签 / 资产分组
      -> 多个分析方向
          -> 数据模型
          -> 维度 / 指标
          -> Metric DSL / 指标语义定义
          -> 主指标集（后台自动生成）
          -> 扩展指标集（可选，系统建议）
          -> 逻辑输出资产
              -> 物理输出资产（可独立，也可共享）
                  -> 计算任务（可独立，也可合并）
```

从资产归属角度看：

```text
主业务主题
  └─ 指标建设方案
      ├─ 场景标签 / 资产分组
      └─ 分析方向
          ├─ 分析方向 1
          │   ├─ 数据模型
          │   ├─ 维度 / 指标
          │   ├─ Metric DSL / 指标语义定义
          │   ├─ 主指标集（后台）
          │   ├─ 扩展指标集（可选）
          │   ├─ 逻辑输出资产
          │   ├─ 物理输出资产
          │   └─ 计算任务
          ├─ 分析方向 2
          │   ├─ 数据模型
          │   ├─ 维度 / 指标
          │   ├─ Metric DSL / 指标语义定义
          │   ├─ 主指标集（后台）
          │   ├─ 逻辑输出资产
          │   ├─ 共享物理输出资产
          │   └─ 共享或合并计算任务
          └─ 分析方向 N
```

## 5. 指标语义层 / Metric DSL

Metric DSL 是 Data Metrics 的机器语义层，也是指标平台的执行协议。页面上的业务口径、字段选择、维度确认和计算配置，最终都必须沉淀为一份可校验、可版本化、可编译 SQL、可生成血缘和可测试的 DSL。

如果没有这层协议，平台只能停留在“确认了中文口径”的产品流程，后台很难稳定生成 SQL、DAG、血缘、质量规则和冲突判断。

### 5.1 DSL 的定位

Metric DSL 承接三类输入：

- 用户通过对话和页面确认的业务口径。
- Data Assets 中已接入的数据模型、字段、关系和质量元数据。
- 系统推荐的指标类型、聚合规则、可加性、粒度和依赖关系。

Metric DSL 输出五类能力：

- 编译 SQL：按查询时计算或离线周期计算生成 SQL。
- 生成 DAG：根据指标依赖、数据模型关系和输出资产生成计算链路。
- 生成血缘：从指标到字段、数据模型、逻辑输出资产、物理输出资产和下游服务形成血缘。
- 生成测试：根据字段、过滤条件、可加性和质量规则生成单元测试、数据测试和新鲜度测试。
- 冲突检测：对名称、业务定义、表达式、字段和过滤条件进行结构化比较。

### 5.2 DSL 核心字段

建议每个指标至少包含以下机器字段：

```yaml
metric:
  name: 成交金额
  code: gmv
  type: atomic
  business_definition: 支付成功且非测试订单的实付金额汇总
  semantic_model: order_payment
  owner: 交易经营部
  version: v1
  status: draft
  measure:
    expr: sum(order_pay_amount)
    agg: sum
    source_field: dwd_payment_detail.order_pay_amount
  filters:
    - pay_status = 'SUCCESS'
    - is_test_order = false
    - order_status != 'CANCELLED'
  time_dimension: pay_time
  grain:
    - date
    - channel
  available_dimensions:
    - date
    - channel
    - region
    - product_category
  additive_type: fully_additive
  time_aggregation: sum_by_day
  default_aggregation: sum
  derived_from: []
  sql_compile:
    engine: spark_sql
    materialization: table
    partition_by: dt
  tests:
    - not_null: order_pay_amount
    - accepted_values: pay_status
    - freshness: pay_time <= T+1
```

其中 `code` 是指标的唯一稳定编码，所有指标依赖、API 查询、SQL 编译、血缘和版本比较都应以 `code + version` 为准，而不是以中文名称为准。

### 5.3 指标类型体系

不同指标类型决定了 SQL 编译、聚合方式、可加性、是否可预计算、是否可参与归因和是否可作为其他指标依赖。

| 指标类型 | 示例 | 特点 | 关键约束 |
| --- | --- | --- | --- |
| 原子指标 | 订单数、成交金额、退款金额 | 直接从事实表聚合 | 通常可离线预计算，可作为派生指标依赖 |
| 派生指标 | 客单价、支付成功率、退款率 | 由一个或多个指标计算 | 不能简单跨维度平均，需保留分子分母或依赖关系 |
| 复合指标 | 渠道贡献率、GMV 下降贡献度 | 依赖多个指标和维度 | 需声明参与计算的维度和归因口径 |
| 比较指标 | 同比、环比、较上周期变化率 | 依赖时间窗口 | 需声明基准窗口、比较窗口和缺失窗口处理 |
| 诊断指标 | 订单量下降贡献、价格带影响 | 面向归因分析 | 需声明拆解框架和贡献度算法 |
| 质量指标 | 空值率、延迟率、重复率 | 面向数据质量 | 依赖元数据和数据测试规则 |
| 运营阈值指标 | 支付成功率低于 95% | 面向监控告警 | 需声明阈值、告警周期和通知策略 |

例如“支付成功率”必须定义为派生比率指标：

```yaml
metric:
  name: 支付成功率
  code: pay_success_rate
  type: derived
  formula: pay_success_order_count / pay_submit_order_count
  numerator: pay_success_order_count
  denominator: pay_submit_order_count
  additive_type: non_additive
  default_aggregation: ratio
```

跨渠道汇总时，系统不能对各渠道支付成功率直接求平均，而必须重新汇总分子和分母：

```text
SUM(pay_success_order_count) / SUM(pay_submit_order_count)
```

### 5.4 可加性、粒度和聚合规则

每个指标必须声明可加性：

| 可加性 | 含义 | 示例 |
| --- | --- | --- |
| 完全可加 | 可按所有已声明维度和时间维度加总 | 成交金额、退款金额 |
| 半可加 | 可按部分维度加总，但不能按某些维度加总 | 库存量、账户余额 |
| 不可加 | 不能直接加总或简单平均 | 客单价、支付成功率、退款率 |

指标还必须声明默认聚合规则：

- `sum`
- `count`
- `count_distinct`
- `avg`
- `ratio`
- `max`
- `min`
- `last_value`

时间聚合规则建议包括：

- `sum_by_day`：按日求和。
- `avg_by_day`：按日均值。
- `period_end`：取期末值。
- `period_max`：取周期最大值。
- `window_compare`：按时间窗口比较。

典型治理规则：

- 成交金额可以按日期、渠道、地区和商品类目加总。
- 订单数可以加总，但 `count_distinct` 类型需要注意跨维度去重。
- 客单价不可加，必须通过成交金额 / 订单数重新计算。
- 支付成功率不可加，必须通过成功订单数 / 提交订单数重新计算。
- 库存量按商品、仓库可加，但按日期不可简单求和，通常取期末值。
- 账户余额按账户可加，但按时间不可加，通常取期末值。

### 5.5 SQL 编译规则

Metric DSL 到 SQL 的编译应分为四步：

1. 解析指标依赖：识别原子指标、派生指标、分子分母、比较窗口和诊断算法。
2. 解析数据模型：根据 `semantic_model`、字段血缘和模型关系确定事实表、维表和 join 路径。
3. 解析粒度和维度：根据 `grain`、`available_dimensions` 和查询需求决定 group by。
4. 生成 SQL 和测试：输出离线结果表 SQL、查询时 SQL、依赖血缘、数据测试和质量规则。

编译器必须阻止以下情况：

- 查询维度不在 `available_dimensions` 内。
- 不可加指标被直接加总。
- 派生比率指标被跨维度简单平均。
- 半可加指标在禁止维度上聚合。
- 派生指标依赖缺失或存在循环依赖。
- 过滤条件引用不存在字段。
- 时间字段与统计窗口不匹配。

### 5.6 版本规则

指标版本建议分为三类：

| 变更类型 | 示例 | 版本策略 |
| --- | --- | --- |
| 文案变更 | 业务说明、负责人、标签 | 小版本更新，不影响历史数据 |
| 兼容口径变更 | 增加可选维度、补充质量测试 | 小版本更新，记录变更说明 |
| 非兼容口径变更 | 过滤条件、时间字段、计算表达式变化 | 新建主版本，需要影响分析、下游通知和历史回刷策略 |

发布时必须保存：

- 当前 DSL 快照。
- 上一版本 DSL 快照。
- 字段级差异。
- SQL 差异。
- 影响的输出资产、计算任务和下游服务。

### 5.7 测试规则

Metric DSL 发布前至少执行以下测试：

- 字段存在性测试。
- 字段类型测试。
- 过滤条件合法性测试。
- 聚合规则测试。
- 可加性测试。
- 分子分母非零测试。
- 数据新鲜度测试。
- 结果波动测试。
- 与存量指标相似度和冲突测试。
- SQL 编译测试。

只有 DSL 校验通过，指标才能进入“可发布”状态。

### 5.8 指标相似度与冲突判断

指标冲突不能只靠名称判断，应同时比较名称、业务定义、计算表达式、来源字段和过滤条件。

建议评分规则：

```text
指标相似度 =
  名称相似度 * 20%
+ 业务定义相似度 * 30%
+ 计算表达式相似度 * 30%
+ 来源字段和过滤条件相似度 * 20%
```

处理策略：

| 相似度 | 处理方式 |
| --- | --- |
| 90% 以上 | 推荐复用 |
| 70% - 90% | 推荐合并或基于已有指标新建版本 |
| 50% - 70% | 提示相似，要求人工确认 |
| 50% 以下 | 允许新建 |

进一步建议把计算表达式和过滤条件解析成标准 AST 后再比较，避免仅靠文本相似度误判。例如：

- 成交金额：支付成功订单实付金额。
- GMV：下单订单应付金额。
- 支付金额：支付流水成功金额。

这三个名称和业务含义接近，但来源字段、过滤条件和时间字段可能不同，必须通过 DSL 结构化比较后才能判断是否复用、合并或新建。

## 6. 核心概念定义

### 6.1 业务问题

用户用自然语言提出的指标建设需求。

示例：

> 最近 30 天成交金额为什么下降？

在 Data Metrics 中，这个问题不是为了立即返回分析结论，而是为了驱动系统生成指标建设方案。

### 6.2 指标建设方案

一次问题驱动生成流程的草案对象，也是流程中最重要的组织载体。

它承接：

- 用户原始问题
- 分阶段澄清记录
- 意图结构化草案
- 口径定义草案
- 推荐主业务主题和关联业务主题
- 推荐依据
- 推荐场景标签
- 推荐分析方向
- 推荐数据模型
- 推荐维度和指标
- 存量资产匹配结果
- 输出资产设计
- 计算任务配置
- 审核意见和版本记录

发布后，方案中的内容会沉淀为正式的业务主题、场景标签、指标集、指标、维度、输出资产和计算任务。

### 6.3 主业务主题和关联业务主题

主业务主题是本次方案的主要归属，也是发布后的管理入口。

关联业务主题用于跨主题复用资产。

示例：

问题：大促 GMV 下降

- 主业务主题：交易经营
- 关联业务主题：营销增长、供应链履约

这样既保持归属清晰，又允许系统复用其他主题下的指标、维度和数据模型。

### 6.4 场景标签 / 资产分组

场景标签是对指标资产的轻量分组，用于检索、治理和复用。

示例：

业务主题：交易经营

场景标签可以包括：

- 经营诊断
- 交易分析
- 大促复盘
- 支付转化
- 订单履约

场景标签解决的是：

> 本次方案沉淀的资产以后应该如何被发现、复用和治理？

指标体系可以作为后台治理概念继续保留，但不建议在问题驱动生成流程中强制用户选择或创建。一个分析方向生成的资产可以同时打多个标签，例如“经营诊断”和“交易分析”。

### 6.5 推荐依据 / 拆解框架

推荐依据是系统把业务问题拆成分析方向的说明层。

示例：

- 因子拆解：成交金额 = 订单量 × 客单价 × 支付转化率
- 漏斗拆解：访问 -> 加购 -> 下单 -> 支付 -> 复购
- 生命周期拆解：新增 -> 活跃 -> 留存 -> 流失 -> 召回
- 贡献拆解：按渠道、地区、商品、客户分层拆解贡献
- 影响因素拆解：退款、缺货、活动、价格、履约等外部扰动

推荐依据解决的是：

> 为什么系统会推荐这些分析方向？

它不应该是一个独立必填步骤，而应该默认折叠展示在分析方向列表上方。用户需要理解推荐逻辑时再展开查看。

### 6.6 分析方向

分析方向是系统对业务问题拆解出的建设方向，也是当前方案里的核心建设边界。

示例：

- 订单趋势分析
- 支付转化分析
- 渠道贡献分析
- 退款影响分析
- 商品结构分析

一个分析方向默认生成一组指标和维度、一个主指标集、一个逻辑输出资产和一个主计算入口。

后台可以根据系统检测结果建议：

- 拆出扩展指标集，用于承载归因、细分或高级消费场景。
- 与其他分析方向共享同一物理输出资产，用于减少重复存储。
- 与其他分析方向共享或合并计算任务，用于减少重复计算。

用户侧需要区分分析方向的优先级：

- 核心方向：直接对应问题的主要驱动因子，默认展开并优先确认。
- 扩展方向：用于补充解释外部扰动，默认折叠，用户按需加入。
- 暂缓方向：相关但当前数据覆盖不足或业务价值不明确，暂不进入本次建设。

### 6.7 数据模型

支撑某个分析方向的数据来源模型。

数据模型来自 Data Assets 中已接入的数据源和资产关系。

推荐数据模型时，不能只展示模型名称，还需要展示：

- 覆盖哪些指标
- 覆盖哪些维度
- 缺失哪些字段
- 字段语义可信度
- 表关系可信度
- 数据新鲜度
- 最近字段变更
- 数据质量风险

### 6.8 指标集

由一个分析方向沉淀出的指标消费单元。

一个指标集包含一组指标和维度，面向一个明确的分析场景。

在用户侧不需要单独创建指标集。用户确认分析方向后，系统后台自动生成主指标集，并在资产详情、治理、血缘和发布记录中展示。

指标集分为两类：

- 主指标集：每个分析方向唯一，承载该方向的核心指标和治理锚点。
- 扩展指标集：可选，由系统建议创建，用于承载明显不同层次或消费场景的指标。

系统建议扩展指标集的典型条件：

- 指标数量较多，且已经出现清晰分层，例如核心监控指标和归因指标。
- 消费场景不同，例如经营看板、问题诊断、归因分析分别需要不同指标组合。
- 维度粒度明显不同，例如日粒度趋势指标和 SKU 粒度归因指标混在一起。
- 指标维护频率不同，例如核心指标稳定，归因指标经常调整。

扩展指标集必须复用主指标集的业务口径，不允许引入与主指标集冲突的指标定义。

示例：

- 订单趋势分析指标集
- 支付转化分析指标集
- 渠道贡献分析指标集

### 6.9 指标和维度

指标用于量化问题，维度用于拆解问题。

指标示例：

- 成交金额
- 订单数
- 客单价
- 支付成功率
- 渠道成交贡献率

维度示例：

- 日期
- 渠道
- 地区
- 商品类目
- 支付方式

维度需要区分公共维度和方向特有维度：

- 公共维度：日期、渠道、地区、客户等级等，应在同一业务主题或场景标签下保持统一口径。
- 方向特有维度：退款类型、价格带、支付失败原因等，只服务于特定分析方向。

### 6.10 输出资产

某个分析方向最终生成的结果资产。

输出资产分为两层：

- 逻辑输出资产：面向用户、血缘、权限、消费说明和影响分析的资产，每个分析方向默认一个。
- 物理输出资产：真实落地的表、视图、API 或缓存，可以被多个逻辑输出资产共享。

当前版本默认物理输出资产为 ADS 结果表，例如：

- `ads_metric_order_trend_daily`
- `ads_metric_payment_conversion_daily`
- `ads_metric_channel_contribution_daily`

系统可以建议多个分析方向共享同一个物理输出资产。例如订单量变化、客单价变化和渠道结构变化都基于订单交易明细、日粒度和渠道粒度，可以共享 `ads_metric_order_daily`，但各自保留独立逻辑输出资产。

未来物理输出资产可以扩展为：

- ADS 结果表
- 查询视图
- API 服务
- BI 虚拟数据集
- 预聚合缓存

### 6.11 计算任务

负责生成物理输出资产的执行任务。

当前版本建议：

```text
一个物理输出资产 -> 一个主计算任务
```

如果多个计算任务的数据源、粒度、调度和质量规则高度重叠，系统可以建议共享或合并计算任务。合并后，物理计算链路可以共用，但各分析方向仍应保留独立任务视角，用于监控、告警、血缘和影响分析。

计算任务需要包含：

- 任务名称
- 计算方式
- 调度周期
- 分区策略
- 回刷策略
- 依赖数据模型
- 物理输出资产
- DAG
- SQL
- 失败重试策略
- 数据质量校验

## 7. 推荐主流程

### 步骤 1：提出业务问题 / 初步澄清

用户提出业务问题，系统先做必要澄清，不要求一次性把所有口径补齐。

初步澄清只聚焦影响整体方案的关键问题：

- 业务目标
- 时间范围
- 核心统计口径
- 分析对象
- 明确已知的排除条件
- 重点关注的渠道、地区、客户或商品
- 用户已经知道的业务背景

示例追问：

- 成交金额是否只统计支付成功订单？
- 是否重点关注某些渠道？
- 这次更关心发现原因，还是沉淀一套长期监控资产？

对于暂时说不清楚的内容，系统可以先标记为“待确认”，后续在分析方向或指标确认时再补充。

### 步骤 2：生成初版意图和口径草案

初步澄清后，系统生成可确认的结构化草案。

示例：

```text
业务目标：分析最近 30 天成交金额下降原因
时间范围：最近 30 天
成交金额口径：支付成功订单金额，不扣除退款
排除条件：取消订单、测试订单（待确认）
重点渠道：小程序、门店
重点维度：渠道、地区、商品类目
```

同时生成口径定义草案：

```text
成交金额 = 支付成功订单的订单实付金额汇总
默认排除取消订单、测试订单，需业务确认
退款金额不从成交金额中扣除，但作为独立影响因素分析
```

用户确认的是“当前可用版本”，不是最终冻结版本。后续如发现某个方向或指标口径不清晰，可以继续补充澄清，并回写到草案中。

### 步骤 3：存量资产感知

系统基于结构化意图扫描已有资产：

- 是否已有相同或相似业务主题
- 是否已有可复用场景标签或资产分组
- 是否已有相同分析方向
- 是否已有可复用指标集或输出资产
- 是否已有相同指标或相似指标
- 是否已有公共维度
- 是否已有可复用输出资产或计算任务

系统需要给出匹配结果和处理建议。

示例：

```text
已发现：
- “订单趋势分析指标集”已存在，覆盖成交金额、订单数、客单价。
- “支付成功率”指标已存在，但口径为“支付成功订单数 / 提交订单数”，与本次推荐一致。
- “渠道”公共维度已存在，可直接复用。
```

可选处理方式：

- 复用已有
- 合并补充
- 基于已有新建版本
- 新建资产
- 标记冲突待审核

### 步骤 4：推荐主业务主题和关联业务主题

系统推荐本次方案的主业务主题和关联业务主题。

示例：

主业务主题：

- 交易经营

关联业务主题：

- 营销增长
- 供应链履约

主业务主题用于资产归属，关联业务主题用于跨主题复用指标、维度和数据模型。

### 步骤 5：确认分析方向

系统直接推荐分析方向，并在列表上方提供可折叠的“推荐依据”。

推荐依据示例：

```text
系统基于以下逻辑推荐分析方向：
成交金额 = 订单量 × 客单价 × 支付转化率

核心方向覆盖主要驱动因子。
扩展方向覆盖退款、渠道结构、商品结构等外部扰动。
```

推荐方向按优先级分组展示。

核心方向：

- 订单量变化分析
- 客单价变化分析
- 支付转化分析

扩展方向：

- 渠道结构变化分析
- 退款影响分析
- 商品结构分析

暂缓方向：

- 库存和履约影响分析，原因是当前数据模型覆盖度不足

每个分析方向需要展示：

- 推荐原因
- 推荐优先级：核心、扩展、暂缓
- 推荐依据摘要
- 是否已有相似指标集
- 建议操作：保留、移除、暂缓、复用、补充、新建
- 是否需要方向级补充澄清

如果某个方向存在关键口径不清晰，系统可以在该方向下发起补充澄清。

示例：

```text
退款影响分析需要确认：
退款是否影响成交金额主口径？
是否需要区分全额退款和部分退款？
```

### 步骤 6：按分析方向推荐数据模型

系统根据每个分析方向推荐支撑数据模型，并展示覆盖度和可信度。

示例：

| 分析方向 | 推荐数据模型 | 覆盖度 | 可信度 | 风险 |
| --- | --- | --- | --- | --- |
| 订单趋势分析 | 订单交易库、商品库存库 | 覆盖成交金额、订单数、商品类目 | 高 | 无明显风险 |
| 支付转化分析 | 支付明细库、订单交易库 | 覆盖支付状态、支付金额、支付方式 | 中 | 支付失败原因字段缺失 |
| 渠道贡献分析 | 订单交易库、营销活动库 | 覆盖渠道、活动、订单金额 | 中 | 营销活动和订单关联关系待确认 |

数据模型推荐必须可解释，避免只展示库表名称。

### 步骤 7：按分析方向确认维度和指标

系统按分析方向分组推荐维度和指标。

#### 订单趋势分析

指标：

- 成交金额
- 订单数
- 客单价

维度：

- 日期
- 渠道
- 地区
- 商品类目

#### 支付转化分析

指标：

- 支付成功率
- 支付失败订单数
- 支付金额

维度：

- 日期
- 支付渠道
- 支付方式

系统需要对每个指标和维度展示：

- 是否已有资产
- 是否可复用
- 是否存在口径冲突
- 是否与已有资产相似
- 指标类型
- 指标唯一编码
- 业务口径
- 计算表达式
- 过滤条件
- 时间字段
- 统计粒度
- 可用维度
- 可加性
- 默认聚合规则
- 时间聚合规则
- 派生指标依赖
- 来源字段
- 推荐原因
- 可信度
- DSL 校验状态
- 相似度评分和处理建议
- 用户操作：确认、编辑、复用、合并、移除
- 是否需要指标级补充澄清

如果用户在这里补充口径，系统需要同步更新意图和口径草案。

示例：

```text
用户在“成交金额”指标中补充：
只统计支付成功订单，不扣除退款，但需要排除测试订单。

系统回写口径草案：
成交金额 = 支付成功且非测试订单的实付金额汇总。

系统同步更新 Metric DSL：

```yaml
metric:
  name: 成交金额
  code: gmv
  type: atomic
  business_definition: 支付成功且非测试订单的实付金额汇总
  filters:
    - pay_status = 'SUCCESS'
    - is_test_order = false
  additive_type: fully_additive
  default_aggregation: sum
```
```

### 步骤 8：确认输出资产和计算任务

系统基于 Metric DSL 编译 SQL、生成逻辑输出资产和主计算入口。主指标集由后台自动生成，不作为用户侧单独确认步骤。

```text
分析方向 -> 逻辑输出资产 -> 物理输出资产 -> 计算任务
```

示例：

| 分析方向 | 逻辑输出资产 | 物理输出资产 | 计算任务 | 后台主指标集 |
| --- | --- | --- | --- | --- |
| 订单量变化分析 | 订单量变化结果 | `ads_metric_order_daily` | `task_metric_order_daily` | 自动生成 |
| 客单价变化分析 | 客单价变化结果 | `ads_metric_order_daily` | `task_metric_order_daily` | 自动生成 |
| 支付转化分析 | 支付转化结果 | `ads_metric_payment_daily` | `task_metric_payment_daily` | 自动生成 |

如果系统检测到多个方向可以共享物理输出资产或计算任务，在本步骤顶部以建议卡片提示：

```text
系统检测到“订单量变化分析”“客单价变化分析”“渠道结构变化分析”
三个方向的数据源、统计粒度和调度周期高度一致，建议共享一张结果表
ads_metric_order_daily，以减少重复计算和存储成本。

[接受建议] [保持独立]
```

用户不需要理解后台物理表、任务合并和血缘维护细节，只需要确认或拒绝建议。

每个计算任务需要展示：

- 覆盖指标
- 支持维度
- Metric DSL 编译摘要
- DSL 校验结果
- 逻辑输出资产
- 物理输出资产
- 计算逻辑 DAG
- 由 DSL 编译生成的 SQL
- 调度周期
- 分区策略
- 数据质量校验
- 可加性和粒度校验
- 派生指标依赖校验

### 步骤 9：确认发布方案

发布前需要展示完整方案：

- 主业务主题
- 关联业务主题
- 场景标签
- 分析方向
- 指标
- 维度
- Metric DSL 快照
- 指标类型和可加性校验结果
- 逻辑输出资产
- 物理输出资产
- 计算任务
- 存量资产复用清单
- 新建资产清单
- 共享资产建议和用户处理结果
- 相似度评分和处理结果
- 口径冲突和待审核项

发布确认页应增加简化的资产结构图，展示分析方向和物理输出资产的映射关系：

```text
分析方向                      物理输出资产
┌─────────────────┐          ┌─────────────────────┐
│ 订单量变化分析   │──────┐   │                     │
├─────────────────┤      │   │  ads_metric_order   │
│ 客单价变化分析   │──────┼───│     _daily          │
├─────────────────┤      │   │   (共享)            │
│ 渠道结构变化分析 │──────┘   │                     │
└─────────────────┘          └─────────────────────┘
┌─────────────────┐          ┌─────────────────────┐

用户确认后进入审核或发布流程。

发布后，系统沉淀正式资产，并进入版本治理。

发布动作包括：

- 发布业务主题关联
- 发布场景标签
- 后台生成主指标集和扩展指标集
- 发布指标和维度
- 发布逻辑输出资产
- 发布物理输出资产
- 发布或合并计算任务
- 生成血缘
- 生成版本记录
- 通知下游消费方

## 8. 资产冲突与复用机制

系统必须避免盲目新建资产。

### 8.1 冲突类型

常见冲突包括：

- 指标集已存在
- 指标名称相同但口径不同
- 指标口径相同但名称不同
- 维度名称相同但来源字段不同
- 逻辑输出资产或物理输出资产已存在但覆盖范围不同
- 计算任务已存在但调度周期不同

### 8.2 冲突判断规则

指标冲突判断应基于 Metric DSL，而不是只看名称。

系统至少比较四层信息：

- 名称相似度：中文名、英文编码、同义词、业务别名。
- 业务定义相似度：业务口径、适用场景、排除条件。
- 计算表达式相似度：聚合函数、公式、分子分母、派生依赖。
- 来源字段和过滤条件相似度：数据模型、字段血缘、时间字段、过滤条件。

建议评分规则：

```text
指标相似度 =
  名称相似度 * 20%
+ 业务定义相似度 * 30%
+ 计算表达式相似度 * 30%
+ 来源字段和过滤条件相似度 * 20%
```

处理策略：

| 相似度 | 推荐处理 |
| --- | --- |
| 90% 以上 | 推荐复用已有指标 |
| 70% - 90% | 推荐合并或基于已有指标新建版本 |
| 50% - 70% | 提示相似，要求人工确认 |
| 50% 以下 | 允许新建 |

### 8.3 处理策略

系统应提供以下操作：

| 场景 | 推荐操作 |
| --- | --- |
| 已有指标完全满足本次需求 | 复用已有 |
| 已有指标集缺少部分指标 | 合并补充 |
| 已有指标口径需要调整 | 基于已有新建版本 |
| 已有指标口径冲突 | 标记冲突待审核 |
| 推荐资产价值低 | 放弃推荐 |

### 8.4 页面提示

示例：

```text
系统发现“订单趋势分析指标集”已存在。

已覆盖：
- 成交金额
- 订单数
- 客单价

缺少：
- 渠道成交贡献率

建议：
- 复用已有指标集，并补充“渠道成交贡献率”
```

## 9. 方案生命周期与协同

指标建设方案必须是可持久化对象，不能只存在于当前页面状态中。

### 9.1 生命周期状态

建议状态：

```text
草稿中 -> 待确认 -> 待审核 -> 已发布 -> 已归档
```

可选异常状态：

```text
已驳回
发布失败
待补充数据模型
待处理冲突
```

### 9.2 协同能力

指标建设方案应支持：

- 保存草稿
- 继续编辑
- 多人协同编辑
- 评论和审核意见
- 版本对比
- 发布前后差异对比
- 回滚或另存版本
- 操作日志

### 9.3 中断恢复

由于问题驱动生成流程较长，用户可能中断操作。

系统必须支持从草稿恢复，并能恢复：

- 原始问题
- 分阶段澄清记录
- 结构化意图
- 已确认业务主题
- 已确认分析方向
- 已确认维度和指标
- 计算任务配置

## 10. 维度与指标治理

### 10.1 公共维度治理

同一业务主题或场景标签下的公共维度必须保持一致。

例如“渠道”不能在不同指标集中分别来自：

- `order.channel`
- `payment.pay_channel`
- `campaign.channel_name`

除非系统明确建立了统一映射关系。

公共维度应沉淀为维度资产，例如：

- 日期维度
- 渠道维度
- 地区维度
- 客户等级维度
- 商品类目维度

### 10.2 方向特有维度

方向特有维度只服务于特定分析方向。

示例：

- 支付转化分析：支付失败原因、支付方式
- 退款影响分析：退款类型、退款原因
- 商品结构分析：价格带、SKU、品牌

### 10.3 指标治理

指标确认时需要展示：

- 指标名称
- 指标类型
- 业务口径
- 计算公式
- 来源字段
- 统计粒度
- 统计周期
- 所属业务主题和场景标签
- 是否已有相似指标
- 是否存在口径冲突

## 11. 数据模型推荐可信度

数据模型推荐必须展示“为什么推荐”和“推荐是否可靠”。

建议展示以下信息：

| 信息 | 说明 |
| --- | --- |
| 覆盖指标 | 该模型可支撑哪些指标 |
| 覆盖维度 | 该模型可支撑哪些维度 |
| 缺失字段 | 缺哪些关键字段 |
| 语义可信度 | 字段语义是否已确认 |
| 关系可信度 | 表之间关联关系是否可靠 |
| 数据新鲜度 | 最近数据更新时间 |
| 数据质量风险 | 空值、重复、延迟、字段变更 |
| 血缘影响 | 使用该模型会影响哪些下游资产 |

示例：

```text
订单交易库
覆盖度：92%
可信度：高

已覆盖：
- 成交金额
- 订单数
- 客单价
- 日期
- 渠道
- 商品类目

风险：
- 最近 7 天无字段变更
- 数据延迟 15 分钟
```

## 12. 输出资产与计算任务

### 12.1 当前版本关系

当前版本建议：

```text
一个分析方向 -> 一个逻辑输出资产
一个逻辑输出资产 -> 一个主物理输出资产，或共享物理输出资产
一个物理输出资产 -> 一个主计算任务，或共享/合并计算任务
```

指标集由后台自动生成，用于资产治理和消费元数据管理。用户侧仍然理解为“每个分析方向有自己的产出”，后台可以通过共享物理资产和合并计算任务提升效率。

### 12.2 输出资产类型

当前版本默认物理输出资产是 ADS 结果表。

未来可扩展为：

- ADS 结果表
- 查询视图
- API 服务
- BI 虚拟数据集
- 预聚合缓存

### 12.3 共享物理输出资产

多个分析方向可以共享同一物理输出资产，但前提是系统检测到它们具有足够高的一致性。

建议判断条件包括：

| 条件 | 说明 |
| --- | --- |
| 数据源重叠 | 源表、关联逻辑和核心过滤条件高度一致 |
| 统计粒度一致 | 时间粒度、业务粒度、维度粒度基本一致 |
| 分区策略一致 | 分区字段、分区周期和保留策略一致 |
| 调度频率一致 | 调度周期和数据新鲜度要求一致 |
| 权限范围一致 | 数据访问权限和敏感字段控制兼容 |
| 质量规则一致 | 空值、重复、波动、延迟等校验规则兼容 |
| 回刷策略一致 | 历史分区覆盖范围和版本保留策略兼容 |
| 变更影响可控 | 任一方向变更不会让共享宽表变成难以维护的公共大表 |

共享后，各分析方向保留独立逻辑输出资产：

```text
订单量变化分析 -> 订单量变化结果 -> ads_metric_order_daily
客单价变化分析 -> 客单价变化结果 -> ads_metric_order_daily
渠道结构变化分析 -> 渠道结构结果 -> ads_metric_order_daily
```

这样既能复用物理表，又能保持方向级血缘、权限、说明和影响分析。

### 12.4 计算任务设计

计算任务应与物理输出资产对应。

计算任务需要包含：

- 输入数据模型
- 物理输出资产
- 计算 SQL
- DAG
- 调度周期
- 分区字段
- 数据质量校验
- 失败重试
- 依赖任务
- 任务负责人

当多个主计算任务存在高度重叠时，系统可以建议共享或合并计算任务，以减少重复计算和存储浪费。共享后，各分析方向仍保留独立任务视角，用于监控、告警和发布影响说明。

### 12.5 扩展指标集建议

扩展指标集不是用户主动配置项，而是系统检测到明显分层后给出的建议。

建议条件包括：

- 核心监控指标和归因诊断指标混在一起，消费场景明显不同。
- 指标数量较多，且可以自然分成趋势、结构、归因、质量等组。
- 指标粒度差异明显，例如日粒度趋势指标和 SKU 粒度归因指标。
- 指标维护频率不同，例如核心指标稳定，归因指标经常调整。

扩展指标集创建后，仍复用主指标集的业务口径和数据模型，不能形成独立冲突口径。

### 12.6 查询时计算扩展

当前版本可以先以离线周期计算为主。

未来如果支持查询时计算，则一个分析方向可以生成查询视图或语义服务配置，不一定生成周期调度任务。

但产品上仍应保持：

```text
分析方向 -> 逻辑输出资产或服务配置
```

### 12.7 用户侧交互原则

扩展和共享能力不应变成新的配置负担。

用户侧遵循三个原则：

- 默认不展示复杂选项：主指标集、逻辑输出资产和主计算入口自动生成。
- 只在系统检测到明显收益时提示建议：扩展指标集、共享物理资产、合并计算任务都以建议卡片出现。
- 只给简单决策：建议卡片提供“接受建议 / 保持独立”，不在生成流程中暴露复杂资产关系配置。

如果用户在生成流程中拒绝共享或合并建议，后续仍可在资产治理页面中重新触发资产合并、任务合并或指标集拆分。

## 13. 口径变更、回刷和下游影响

指标资产发布后，口径可能被修正。

例如下游 Data Analytics 发现某指标口径错误，需要反向触发变更。

### 13.1 变更类型

常见变更包括：

- 指标公式变更
- 过滤条件变更
- 统计窗口变更
- 维度映射变更
- 来源字段变更
- 逻辑输出资产或物理输出资产结构变更
- 调度周期变更

### 13.2 变更影响分析

系统需要识别：

- 影响哪些指标
- 影响哪些指标集
- 影响哪些逻辑输出资产和物理输出资产
- 影响哪些计算任务
- 影响哪些看板、API、问数服务和分析任务
- 是否需要重跑历史分区

### 13.3 历史数据回刷

当口径变更影响历史结果时，需要支持回刷策略：

- 不回刷，仅从新分区生效
- 回刷指定时间范围
- 全量回刷
- 新旧版本并行保留

计算任务需要支持分区覆盖策略，例如：

```text
回刷最近 30 天分区
回刷 2026-05-01 至 2026-05-29 分区
保留旧版本输出资产 30 天
```

## 14. 异常流程

### 14.1 数据模型不足

如果现有数据模型无法支撑分析方向，系统应提示：

- 缺少哪些字段
- 缺少哪些关联关系
- 建议补充哪个数据源
- 是否暂时移除该分析方向

### 14.2 存量资产冲突

如果发现资产冲突，系统应阻止直接发布，并要求处理：

- 复用已有
- 合并补充
- 新建版本
- 标记待审核

### 14.3 AI 推荐置信度低

如果业务主题、分析方向或数据模型推荐置信度低，系统应展示原因，并要求人工确认。

### 14.4 审核驳回

审核驳回后，方案应回到“待修改”状态，并保留审核意见。

### 14.5 计算任务失败

计算任务失败后，应能定位：

- 输入数据是否延迟
- SQL 是否异常
- 权限是否不足
- 输出表是否冲突
- 数据质量校验是否失败

## 15. 页面流程建议

基于上述方案，Data Metrics 的问题驱动生成流程建议调整为：

```text
1. 提出业务问题
2. 确认初版意图和口径草案
3. 确认主业务主题和关联业务主题
4. 确认分析方向
5. 推荐数据模型
6. 确认维度和指标
7. 确认输出资产和计算任务
8. 确认发布方案
```

页面层面建议：

- 第 2 步展示结构化意图和口径定义文档草案。
- 第 4 步优先展示核心方向，扩展方向折叠展示；推荐依据默认折叠在方向列表上方。
- 第 4 步和第 6 步允许方向级、指标级补充澄清，并回写意图草案。
- 第 5 步按分析方向展示数据模型覆盖度和可信度。
- 第 6 步按分析方向确认维度和指标，同时处理复用、合并、版本和冲突。
- 第 7 步按分析方向展示逻辑输出资产、物理输出资产、DAG、SQL、调度和计算任务。
- 第 7 步如检测到共享物理输出资产或合并计算任务的收益，以建议卡片提供“接受建议 / 保持独立”。
- 第 8 步展示本次新建资产、复用资产、共享资产、冲突项、审核项和发布影响。

## 17. 完整示例

用户提出问题：

> 最近 30 天成交金额为什么下降？

系统澄清后确定：

- 只统计支付成功订单
- 不扣除退款
- 排除取消订单和测试订单
- 重点关注小程序和门店渠道
- 需要按渠道、地区、商品类目拆解

意图结构化草案：

```text
业务目标：分析成交金额下降原因
时间范围：最近 30 天
成交金额口径：支付成功订单金额，不扣除退款
排除条件：取消订单、测试订单

## 18. 一句话总结

Data Metrics 的问题驱动流程应该是：

> 用户提出业务问题，系统先生成可持续完善的意图和口径草案，再推荐主业务主题、关联业务主题和分析方向；推荐依据默认折叠展示，核心方向优先确认，扩展方向按需加入，最终按分析方向确认数据模型、维度指标、逻辑输出资产和计算配置，并在后台自动沉淀主指标集、扩展指标集、共享物理资产、标签、版本和血缘。
